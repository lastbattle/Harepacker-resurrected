# MapSimulator Client Parity Backlog

## Purpose

This document is the single source of truth for MapSimulator parity work against the MapleStory client.

It does three things that the old notes did not do well:

1. Separates shipped work from missing work.
2. Distinguishes broad simulator features from player/avatar parity.
3. Anchors claims to actual code seams and, where relevant, client-side reverse engineering.

## Client Function Index By Backlog Area

The list below maps each backlog area to concrete client seams confirmed in IDA.
These are the first functions to inspect before changing simulator behavior.

### 1. Avatar and player parity

- `CUser::LoadLayer` at `0x8e96d0`
- `CUser::PrepareActionLayer` at `0x8e3070`
- `CUser::SetMoveAction` at `0x8df540`
- `CUser::Update` at `0x8fb8d0`
- `CUser::UpdateMoreWildEffect` at `0x8fb4c0`
- `CUser::SetActivePet` at `0x8e40e0`
- `CUserPool::OnPacket` at `0x94ddf0`
- `CUserPool::Update` at `0x94c370`
- `CUserRemote::Update` at `0x955d50`
- `CAnimationDisplayer::Effect_ByItem` at `0x44f330`
- `CActionMan::GetCharacterImgEntry` at `0x417fd0`
- `CActionMan::LoadTamingMobAction` at `0x427a10`
- `CUserLocal::SetMoveAction` at `0x903ce0`
- `CUserLocal::Update` at `0x937330`

Notes:
`CUser::LoadLayer` and `CUser::PrepareActionLayer` are the clearest resolved avatar-composition seams for layer loading and action-layer setup. `CUser::Update` and `CUser::UpdateMoreWildEffect` also show that the client maintains several additional avatar/effect layers outside the base assembler path, while `CUserLocal::Update` contains the local-player emotion reset path and action/state transitions that matter for expression and rare-action parity. A later IDA pass also confirms that remote avatar ownership is a first-class client surface rather than a loose extension of local-player logic: `CUserPool::OnPacket` dispatches enter, leave, common, remote, and local user packet families, `CUserPool::Update` advances local plus remote users and shared social-effect state, `CUserRemote::Update` owns remote move-action, effect, portable-chair, and prepared-skill presentation, and `CAnimationDisplayer::Effect_ByItem` confirms that one-shot item effects are managed by a dedicated client animation-owner seam rather than only by feature-local overlay code.

### 2. Packet-owned avatar emotion parity discovered by direct `CUserLocal::OnPacket` scan

- `CUser::OnEmotion` at `0x8e0150`
- `CUser::OnRandomEmotion` at `0x8e34b0`

Notes:
The latest direct `CUserLocal::OnPacket` re-check exposed another local-avatar owner that the backlog was not naming anywhere else. `CUser::OnEmotion` decodes an emotion id, duration, and the `m_bEmotionByItemOption` flag before calling `CAvatar::SetEmotion`, while `CUser::OnRandomEmotion` decodes an area-buff item id, resolves `CItemInfo::GetAreaBuffItem`, chooses one weighted emotion slot from the authored prop table, and then routes back through the ordinary emotion-change path via `CWvsContext::SendEmotionChange`. That means client expression parity is not only the passive blink/hit reset surface already tracked in `CUserLocal::Update`; it also includes a packet-owned timed-emotion seam and an item-backed random-emotion owner that should stay explicit in the avatar backlog instead of being hidden under generic item-effect or chat feedback rows.

### 3. Taming-mob action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadTamingMobAction` at `0x427a10`

Notes:
The mount row already cited `LoadTamingMobAction` as evidence, but the owner itself was still buried inside a broader render/state row unlike the explicit loader sections already tracked for summons, mobs, pets, employees, and NPCs. IDA confirms this is a distinct client seam: the loader builds taming-mob action frames only after a vehicle id is already known, and the nearby `IsAbleTamingMobAction` / `IsAbleTamingMobOneTimeAction` gates keep the extra Mechanic and ride coverage keyed to vehicle ownership rather than raw action-name inference alone. That means mounted avatar parity is still split the same way as the other actor families in this backlog set: one row for runtime state/ownership and another for the loader that decides which taming-mob action data the runtime can actually consume.

### 4. Character image-entry ownership discovered by targeted `CActionMan` scan

- `CActionMan::GetCharacterImgEntry` at `0x417fd0`

Notes:
The broader avatar row already cited the runtime owners and later frame-build owners, but another client seam still sat unnamed underneath them. `CActionMan::GetCharacterImgEntry` is not just a raw IMG fetch. The decompile shows it caching by equip UOL key, resolving the formatted character data path, reading `info/islot`, `info/vslot`, `info/sfx`, the weekly-variant flag, and the weapon-side `afterImage`, `walk`, `stand`, `attack`, and `attackSpeed` metadata, then switching weekly-capable equips onto a day-specific property path while keeping the non-weekly fallback image entry alive. That means visible avatar parity is split one layer earlier than the current document said: runtime/state ownership in `CUser::*`, image-entry normalization and metadata ownership in `GetCharacterImgEntry`, frame-build ownership in `LoadCharacterAction`, and then feature-local consumers such as face-look or afterimage work that depend on the normalized character entry.

### 5. Character action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadCharacterAction` at `0x427d00`

Notes:
The broader avatar row already cites `CUser::LoadLayer` and `CUser::PrepareActionLayer`, but a targeted `CActionMan` pass shows there is still another distinct client owner underneath those runtime seams. `CActionMan::LoadCharacterAction` does not only fetch the current action frames. The decompile shows it first normalizing the avatar-hair-equip array for the requested action, zeroing body-part lanes the client refuses to merge, falling back to client default top or bottom items when the incoming equip combination would leave the body invalid, stripping known blocked accessory and weapon-sticker ids, suppressing shield and sticker lanes for ride or vehicle families, applying ghost-mode filtering, and only then calling the lower `load_character_action` plus `MergeCharacterSprite` path. That means visible avatar parity is still split three ways rather than two: runtime ownership in `CUser::*`, packet or state ownership in the local or remote actor seams, and a dedicated character frame-builder that decides which merged body-part set the runtime is even allowed to render for a given action.

### 6. Face-look loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadFaceLook` at `0x41cab0`

Notes:
The existing avatar backlog tracks blink, hit-face, and packet-owned emotion timing, but the loader that actually builds those face-expression canvases was still unnamed. IDA shows `CActionMan::LoadFaceLook` is a dedicated client seam: it caches by skin, face, emotion, and accessory state; resolves the face and accessory image entries; enumerates the authored emotion canvases; reads origin and brow metadata; composites face and accessory planes into the final brow canvas list; and records the authored duration on the cached face-look entry. That means facial-expression parity is not only a runtime question in `PlayerCharacter` or `CharacterAssembler`; it also depends on the client loader that decides which face and accessory canvases exist, how they are composited, and how long the resulting emotion entry stays valid.

### 7. Morph action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::GetMorphImgEntry` at `0x418b30`
- `CActionMan::LoadMorphAction` at `0x4193c0`

Notes:
The avatar backlog already tracks morph runtime and resolver parity, but the underlying morph loader was still folded into that broader row unlike the explicit loader sections already called out for character, face, taming-mob, summon, mob, NPC, pet, employee, and dragon paths. IDA shows the morph path is its own client owner: `CActionMan::GetMorphImgEntry` caches per-template morph IMG entries, resolves the formatted `Morph/<id>.img` resource, follows `info/link` indirection by reloading the linked template and merging missing action branches onto the current morph image entry, and stores the normalized property tree that later action requests consume. `CActionMan::LoadMorphAction` then caches by `(templateId, action)`, resolves the action through the client `s_sMorphAction` table rather than ad hoc string guesses, enumerates frame canvases, reads per-frame `delay`, `lt`, `rb`, and `head` metadata, preserves the authored frame-uol string used by the client morph path, and appends the interior-frame reverse pass when the morph action publishes the replay flag. That means morph parity is still split the same way as the other actor families in this backlog set: runtime transform ownership in the player/avatar seams, plus a dedicated morph image and action loader that decides which morph resources and frame tables the runtime is actually allowed to render.

### 8. Avatar action-layer coordinator ownership discovered by follow-up IDA function scan

- `CAvatar::PrepareActionLayer` at `0x466cb0`
- `CAvatar::PrepareCharacterActionLayer` at `0x465100`
- `CAvatar::PrepareTamingMobActionLayer` at `0x465ce0`
- `CAvatar::FixCharacterPosition` at `0x4616b0`
- `CAvatar::CharacterFrameUpdate` at `0x46abe0`
- `CAvatar::TamingMobFrameUpdate` at `0x46b450`
- `CAvatar::SetMorphed` at `0x469c80`
- `CAvatar::SetRidingVehicle` at `0x46a790`

Notes:
The latest IDA function scan did not find another missing `CActionMan` loader family; those are already distributed across the avatar, skill, combat, NPC, and companion backlog files. It did expose a lower avatar-runtime owner that the avatar document only implied through `CUser::*` and loader rows. `CAvatar::PrepareActionLayer` is the coordinator that resolves move or one-time action into raw action, routes morph templates to `LoadMorphAction`, routes normal and mounted avatars to `LoadCharacterAction` and `LoadTamingMobAction`, clamps frame delay through walk speed or action speed, applies held shoot-action infinite delays, inserts canvases into the under-face, under-character, over-character, and taming-mob layers, and calls `FixCharacterPosition` when riding composition needs the character and taming-mob origins reconciled. `FixCharacterPosition` then derives body, face, muzzle, taming-mob navel, head, and muzzle offsets from the current character and taming-mob frame tables while applying flip-specific signs. That means the remaining visible avatar gap is not only loader data selection or high-level `CUser` state: it also includes the lower runtime layer coordinator that binds those loader outputs into GR2 layer ownership, frame timing, mounted anchor relocation, and morph/taming-mob transition behavior.

## Current State Summary

MapSimulator is no longer "missing player simulation".
The project already has:

- A playable local character with foothold, ladder, rope, swim, fly, jump, prone, hit, death, and respawn states.
- Character WZ loading for body, head, face, hair, equipment, weapon overlays, z-map ordering, and slot-visibility conflicts.
- A working generic skill system with job skill loading, buffs, cooldowns, projectiles, hit effects, and quick-slot assignment.
- Mob and drop pools, hidden portal support, damage number rendering from WZ assets, mob HP bars, and boss HP bars.
- Several field-effect systems and special field implementations that are already present in the current simulator.

The real parity gap is no longer "does the simulator have a feature at all?".
The gap is now "how closely does the current implementation match client behavior, data selection, timing, and fallback rules?".

## Recent Progress Summary

Recent implementation progress has been folded into the backlog tables below instead of being tracked as a separate changelog.
The main baseline updates worth keeping visible are:

- Avatar parity moved out of the stub phase: action coverage, rare-action rendering, facial expression behavior, and anchor fallback rules are implemented, while mount and transform handling is partially in place.
- Skill parity is broader than the old notes implied: the simulator now loads and casts the full player skill catalog, supports runtime job swaps, shows buff and cooldown feedback, and covers more movement-family and summon behavior.
- Physics and movement now have real runtime seams for ladder lookup, float collision, movement-path snapshots, and moving-platform foothold sync.
- Combat and UI both shifted from "missing feature" to "refinement": HP indicators, damage rendering, status-bar layout, warning flashes, chat strip behavior, skill UI layout, and tooltip reuse all exist but still need client-accurate polish.
- NPC and quest interaction now have a usable simulator baseline through dialogue overlays, quest lists, and common quest-state mutations, but full scripting and inventory-backed requirements remain incomplete.
