## Purpose

This document is the single source of truth for MapSimulator parity work against the MapleStory client.

It does three things that the old notes did not do well:

1. Separates shipped work from missing work.
2. Distinguishes broad simulator features from player/avatar parity.
3. Anchors claims to actual code seams and, where relevant, client-side reverse engineering.

Notes:
The companion/social backlog was still missing the packet owner that actually feeds employee actors. IDA decompilation now makes that lifecycle concrete. `CEmployeePool::OnPacket` is a narrow dispatcher for `319` enter-field, `320` leave-field, and `321` mini-room balloon updates. `OnEmployeeEnterField` decodes the employer id, reads the employee template id, creates a `CEmployee`, stores it in the pool map keyed by employer, and then calls both `CEmployee::Init` and `CEmployee::SetBalloon` on the packet payload. `OnEmployeeLeaveField` clears the same pooled actor and removes it once no flags remain. `CEmployee::Init` also confirms that the actor is not just a room-side stub: it decodes world position and foothold, creates its own `CVecCtrl`, builds the action layer, formats the name tag, and prepares the first action layer before `CEmployee::Update` takes over. ### 3. Pet and employee image-entry ownership discovered by targeted `CActionMan` scan

- `CActionMan::GetPetImgEntry` at `0x41b0d0`
- `CActionMan::GetEmployeeImgEntry` at `0x41b490`

Notes:
The same `CActionMan` pass shows the client keeps a template-image owner below the pet and employee action loaders. `GetPetImgEntry` caches by pet template id, resolves the formatted pet IMG path through the resource manager, and falls back to `Effect/PetEff.img/Basic` when no concrete pet template is active before wear-aware action loading begins. `GetEmployeeImgEntry` is the matching hired-merchant/employee-side template seam: it caches by employee template id and resolves the formatted employee IMG property tree before `LoadEmployeeAction` builds any frame table. That means companion parity is split three ways for these actors: packet/runtime ownership, image-entry ownership, and then action loading.

### 4. Pet and employee action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadPetAction` at `0x4213f0`
- `CActionMan::LoadEmployeeAction` at `0x422940`

Notes:
The follow-up `CActionMan` lookup exposed another client-owned companion seam that the existing rows were still flattening into generic runtime update behavior. `CActionMan::LoadPetAction` is not just a frame fetch helper: it caches by pet template and action, uses a fixed base-action table before falling back to `CPetTemplate::GetActionName`, resolves redirected pet action nodes, and composes per-frame timing plus pet-wear overlay data keyed by `nPetWearID`. `CActionMan::LoadEmployeeAction` is the matching hired-merchant/employee-side loader: it caches by template and action, enumerates employee template frames directly, and applies the client default delay of `180` when a frame omits an explicit value. That means companion parity is still missing a named action-loader owner for both pets and employees instead of only needing more `CPet::Update` or `CEmployee::Update` polish.

### 5. Dragon action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadDragonAction` at `0x4247c0`

Notes:
The follow-up `CActionMan` scan exposed another explicit companion loader owner that the backlog was still folding into the broader dragon runtime row. `CActionMan::LoadDragonAction` keeps its own cache keyed by Evan job and action, resolves a formatted `Skill/Dragon/<job>.img` path before looking up the action node, uses the dedicated `s_sDragonAction` action-name table instead of generic avatar-action inference, and builds a full per-frame entry list from the action canvases rather than leaving dragon playback to an ad hoc runtime-only fetch. That means dragon parity should stay split two ways here rather than three: one row for the dragon runtime and one row for the loader that decides which dragon action data exists and how those frames are built.
