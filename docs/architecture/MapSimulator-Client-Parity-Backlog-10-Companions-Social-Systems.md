Notes:
The companion/social backlog was still missing the packet owner that actually feeds employee actors. IDA decompilation now makes that lifecycle concrete. `CEmployeePool::OnPacket` is a narrow dispatcher for `319` enter-field, `320` leave-field, and `321` mini-room balloon updates. `OnEmployeeEnterField` decodes the employer id, reads the employee template id, creates a `CEmployee`, stores it in the pool map keyed by employer, and then calls both `CEmployee::Init` and `CEmployee::SetBalloon` on the packet payload. `OnEmployeeLeaveField` clears the same pooled actor and removes it once no flags remain. `CEmployee::Init` also confirms that the actor is not just a room-side stub: it decodes world position and foothold, creates its own `CVecCtrl`, builds the action layer, formats the name tag, and prepares the first action layer before `CEmployee::Update` takes over. ### 3. Pet and employee action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadPetAction` at `0x4213f0`
- `CActionMan::LoadEmployeeAction` at `0x422940`

Notes:
The follow-up `CActionMan` lookup exposed another client-owned companion seam that the existing rows were still flattening into generic runtime update behavior. `CActionMan::LoadPetAction` is not just a frame fetch helper: it caches by pet template and action, uses a fixed base-action table before falling back to `CPetTemplate::GetActionName`, resolves redirected pet action nodes, and composes per-frame timing plus pet-wear overlay data keyed by `nPetWearID`. `CActionMan::LoadEmployeeAction` is the matching hired-merchant/employee-side loader: it caches by template and action, enumerates employee template frames directly, and applies the client default delay of `180` when a frame omits an explicit value. That means companion parity is still missing a named action-loader owner for both pets and employees instead of only needing more `CPet::Update` or `CEmployee::Update` polish.

## Current State Summary

This area is not completely blank in the simulator, but it is only represented by fragments today.
The project currently includes:

- Pet drop chasing and pickup helpers in `DropPool.cs`, explicitly annotated against `CDropPool::TryPickUpDropByPet`.
- A simulator-owned pet runtime base in `PlayerManager` and `HaCreator/MapSimulator/Companions/`, including an ordered active-pet roster, live pet update loop, WZ-backed pet frame loading, slot-aware ladder or rope back-hang anchors, split behind or in-front pet rendering passes, and the client-confirmed pickup-forbidden gate for map `209080000`.
- Legacy and Big Bang equipment windows that already load pet, dragon, and android tab buttons and slot layouts.
- Sit-state animation keys and install-item drop categorization, which means chair-facing behavior is not entirely absent from the runtime vocabulary.
- Cash-shop-map gating flags in `MapSimulator.cs` and `GameStateManager.cs`, which indicates the simulator already distinguishes that map family even though it does not simulate the broader client surfaces there.

The gap is not whether these systems are acknowledged at all.
The gap is that the simulator does not yet reproduce the client's actual companion runtime, social windows, room systems, storage flows, or chair-backed avatar layering.
