1. Separates shipped work from missing work.
2. Distinguishes broad simulator features from player/avatar parity.
3. Anchors claims to actual code seams and, where relevant, client-side reverse engineering.

### 2. Summoned action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::LoadSummonedAction` at `0x423100`
- `CActionMan::GetSummonedProp` at `0x41b830`

Notes:
The earlier backlog text tracked summon packet ownership and runtime behavior, but the targeted `CActionMan` scan shows another concrete client owner that was still missing from the document: summon action loading itself. `GetSummonedProp` resolves the summon UOL from `SKILLENTRY::GetSummonedUOL` through the resource manager, while `LoadSummonedAction` uses that property to resolve the action-specific node, enumerate each frame canvas, read per-frame `delay`, `z`, `origin`, and `rb` metadata, compute the frame bounds, and cache the assembled action entry under the summon action key. The same decompile also shows a distinct reverse-playback seam: when the source action node publishes the reverse flag, the client appends a reversed copy of the built frame list instead of leaving playback order entirely to the runtime. That means summon parity is not only `CSummoned::Update` timing and `CSummonedPool::OnPacket` ownership; it also depends on the loader that chooses the summon asset path, builds the frame table, and decides whether the runtime receives a forward-only or mirrored playback list.

### 3. Shadow-partner and mirror helper parity discovered by targeted IDA re-check

- `CUser::PrepareActionLayer` at `0x8e3070`
- `CUser::Update` at `0x8fb8d0`

Notes:
This row remains `Partial`. The targeted re-check surfaced a client-owned helper seam that the backlog was not tracking explicitly, and the simulator now mirrors more of that ownership split instead of treating every clone-family helper as the same WZ-backed shadow-partner path. WZ was rechecked first for `skill/433.img/skill/4331002`: it still exposes `action/0 = alert` plus buff and effect data but no `special/*` helper action branch, which keeps `Mirror Image` (`4331002`) on the assembled-avatar helper seam while Shadow Partner continues to use the WZ-authored helper action catalog. Client evidence stayed second: `CUser::PrepareActionLayer` still calls `CUser::PrepareShadowPartnerActionLayer` and `CUser::PrepareMirrorActionLayer` immediately after the main avatar action layer is prepared; `CUser::GetMirrorSrcLayer` still returns the five avatar layer handles in fixed index order (`UnderCharacter`, `OverCharacter`, `UnderFace`, `Face`, `OverFace`); and `CUser::PrepareMirrorActionLayer` still recreates a mirror helper layer whenever that helper layer is missing or its flip diverges from `m_pLayerUnderFace`, then re-parents every recreated mirror plane against `m_pLayerUnderFace`, copies origin from the live source layer, seeds the 200 ms `RelMove` from the live source-layer current time, and uses the live source-layer canvas instead of a per-piece redraw.

The simulator already kept the earlier landed ownership split and render seam for this row, and this pass narrows the remaining mismatch again: `SkillData` / `SkillManager` still recognize `Mirror Image` as a distinct helper owner, `PlayerCharacter` still keeps mirror runtime state separate from the `special/*` Shadow Partner actor, mirror suppression still follows the client-backed raw-action and state gates, and the dedicated prepared five-layer cache still inserts the full mirror package at the shared UnderFace overlay boundary. This pass now also makes the prepared/live reuse gate match the newly rechecked client recreate seam more closely: live mirror-source reuse no longer survives action-name drift, frame drift, or flip-only drift, so the simulator stops treating those cases as silently reusable prepared mirror planes when the client would recreate or re-seed the helper layer instead. The remaining gap is narrower and still concentrated at the real live-layer-object boundary inside `CUser::PrepareMirrorActionLayer` / `CUser::Update`: the simulator still does not own shared `IWzGr2DLayer` mirror objects, exact live-layer origin and current-time snapshots, true `InsertCanvas` reparenting against the client layer graph, or the follow-up helper behaviors that depend on those live client layer objects rather than simulator-owned prepared plane snapshots.

### 4. Shadow-partner action-loader ownership discovered by targeted `CActionMan` scan

- `CActionMan::GetShadowPartnerProp` at `0x41ba60`
- `CActionMan::LoadShadowPartnerAction` at `0x423cf0`

Notes:
The current skill backlog already names the shadow-partner runtime/helper surface, but the targeted `CActionMan` pass shows there is still a distinct loader owner underneath it. `GetShadowPartnerProp` resolves the helper property from the skill id before action lookup, while `LoadShadowPartnerAction` caches by `(skillId, action)`, resolves either the named raw-action branch or the numeric fallback branch through `get_action_name_from_code`, enumerates the action canvases into cached frame entries, reads frame-owned metadata from the canvas properties, and appends the mirrored interior-frame tail when the loader's replay table marks that action. That means shadow-partner parity is split the same way as summons, pets, employees, dragons, mobs, NPCs, and taming-mob helpers elsewhere in the backlog set: `CUser::PrepareShadowPartnerActionLayer` and the simulator helper runtime own placement and playback state, while `CActionMan` owns the helper action catalog and the frame table those runtime paths consume.

### 6. Packet-owned combo counter and combo-command parity discovered by direct `CUserLocal::OnPacket` scan

### 7. Shared skill-use animation-owner parity discovered by targeted `CAnimationDisplayer` scan

- `CAnimationDisplayer::Effect_SkillUse` at `0x459100`

Notes:
The broader skill rows already cover cast logic, hit effects, prepared gauges, and helper actors, but the latest `CAnimationDisplayer` pass exposed another owner that the backlog corpus was not naming anywhere else. `Effect_SkillUse` is not a generic data lookup. The same body also keeps optional overlay-parent and auto-flip ownership at this animation-manager seam, which means visible skill-cast parity is split one level further than the current skill backlog suggests: `CUserLocal::*` owns cast admission and follow-up behavior, feature-local runtime code owns gameplay consequences, and `CAnimationDisplayer::Effect_SkillUse` owns the shared one-shot cast animation composition that many skills route through before those consequences are visible.

## Current State Summary

MapSimulator is no longer "missing player simulation".
The project already has:

- A playable local character with foothold, ladder, rope, swim, fly, jump, prone, hit, death, and respawn states.
- Character WZ loading for body, head, face, hair, equipment, weapon overlays, z-map ordering, and slot-visibility conflicts.
- A working generic skill system with job skill loading, buffs, cooldowns, projectiles, hit effects, and quick-slot assignment.
- Mob and drop pools, hidden portal support, damage number rendering from WZ assets, mob HP bars, and boss HP bars.
- Several field-effect systems and special field implementations that are already present in the current simulator.

The real parity gap is no longer "does the simulator have a feature at all?".
The gap is now "how closely does the current implementation match client behavior, data selection, timing, and fallback rules?".

## Recent Progress Summary

Recent implementation progress has been folded into the backlog tables below instead of being tracked as a separate changelog.
The main baseline updates worth keeping visible are:

- Avatar parity moved out of the stub phase: action coverage, rare-action rendering, facial expression behavior, and anchor fallback rules are implemented, while mount and transform handling is partially in place.
- Skill parity is broader than the old notes implied: the simulator now loads and casts the full player skill catalog, supports runtime job swaps, shows buff and cooldown feedback, and covers more movement-family and summon behavior.
- Physics and movement now have real runtime seams for ladder lookup, float collision, movement-path snapshots, and moving-platform foothold sync.
- Combat and UI both shifted from "missing feature" to "refinement": HP indicators, damage rendering, status-bar layout, warning flashes, chat strip behavior, skill UI layout, and tooltip reuse all exist but still need client-accurate polish.
- NPC and quest interaction now have a usable simulator baseline through dialogue overlays, quest lists, and common quest-state mutations, but full scripting and inventory-backed requirements remain incomplete.
