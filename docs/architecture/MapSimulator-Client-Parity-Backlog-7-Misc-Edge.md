## Purpose

This document is the single source of truth for MapSimulator parity work against the MapleStory client.

It does three things that the old notes did not do well:

1. Separates shipped work from missing work.
2. Distinguishes broad simulator features from player/avatar parity.
3. Anchors claims to actual code seams and, where relevant, client-side reverse engineering.

Notes:
`CUserLocal::TalkToNpc` is the first concrete client seam for future NPC bridge work. `CUserLocal::Update` also shows client-side quest-alert refresh calls, emotion reset / balloon lifetime upkeep, timer refresh, timed direction-mode release, and status-bar chat logging, while `CUserLocal::TryAutoRequestFollowCharacter` covers automatic escort/follow requests and `CUser::PetAutoSpeaking` covers one of the idle-feedback surfaces tied to those broader interaction loops. A follow-up IDA scan also showed that reactor ownership is split across both local collision and a packet-owned pool: `CReactorPool::OnPacket` dispatches reactor change-state (`334`), move (`335`), enter-field (`336`), and leave-field (`337`) traffic into dedicated handlers instead of leaving all reactor lifecycle behavior on `CUserLocal::CheckReactor_Collision`.

### 2. Reactor layer-build and hit-prop ownership discovered by targeted `CReactorPool` decompile

- `CReactorPool::LoadReactorLayer` at `0x6ce630`

Notes:
The latest targeted decompile exposes a distinct reactor owner that should not stay buried inside the broader packet-lifecycle row. `CReactorPool::LoadReactorLayer` lazy-resolves `CReactorTemplate`, looks up the current state branch by stringified state id, builds the visible state layer through `CAnimationDisplayer::LoadLayer`, switches between repeat and stop playback from `CReactorTemplate::STATEINFO::m_bRepeat`, preserves timed movement against `tStateEnd` for moving states, auto-selects hit reactions through `get_hittype_priority_level` plus `CReactorTemplate::GetHitProp` when no proper event index has been fixed yet, formats and plays the per-state reactor hit sound through the StringPool template at id `2121`, and then assigns the final layer depth from the closest foothold page or z-mass plus the reactor template layer mode. That makes reactor parity split the same way as the other action-loader families already called out elsewhere in the backlog set: `CReactorPool::OnPacket` owns lifecycle and timing, while `LoadReactorLayer` owns state-layer rebuild, hit-prop arbitration, sound, and final z placement.

### 3. NPC action-loader and client-action-set parity discovered by targeted `CActionMan` scan

- `CActionMan::LoadNpcAction` at `0x420ae0`

Notes:
The targeted `CActionMan` lookup exposed another NPC-side owner that was still only implied by broader talk or actor rows. `CActionMan::LoadNpcAction` caches by NPC template, action, and client action-set index; it also has a special `nClientActionIdx == -2` branch that calls `CNpcTemplate::CalcClientActionSetIdx` before choosing the action family, applies the client default frame delay of `180` when delay data is absent, and can append reversed frames when the authored data marks that path. That means NPC parity is still missing a dedicated action-loader seam for client action-set selection and frame-table construction instead of only needing more dialogue or field-placement polish.

### 4. NPC image-entry ownership discovered by targeted `CActionMan` scan

- `CActionMan::GetNpcImgEntry` at `0x41a840`

Notes:
The same `CActionMan` pass shows the client keeps another NPC-side owner below `LoadNpcAction`: template-image resolution itself. `GetNpcImgEntry` caches by NPC template id, resolves the formatted NPC resource path through the resource manager, checks the template `link` field, recursively reopens linked NPC templates, and copies any missing top-level branches from the linked image onto the cached property tree before client-action-set selection runs. That means NPC parity is not only talk flow plus action loading; it also depends on the earlier image-entry seam that decides which normalized NPC IMG entry the action-set loader sees.

### 5. Packet-authored direction and stand-alone mode parity discovered by direct `CUserLocal::OnPacket` scan

- `CUserLocal::OnSetDirectionMode` at `0x9054f0`
- `CUserLocal::OnSetStandAloneMode` at `0x905550`

Notes:
The existing backlog tracks delayed `TryLeaveDirectionMode` release, but the direct `CUserLocal::OnPacket` re-check exposed the client-owned state-entry seam that actually drives those releases. `OnSetDirectionMode` decodes an enable byte plus a delay, clears `m_bAfterLeaveDirectionMode`, enters direction mode immediately when the packet enables it or when the delay is non-positive, and otherwise arms a delayed leave by storing `m_tAfterLeaveDirectionMode = delay + get_update_time()`. `OnSetStandAloneMode` is a separate packet-owned toggle that writes the decoded byte into the `CWvsContext` stand-alone flag rather than behaving like another window-local direction-mode alias. That owner pair is not named anywhere else in the current backlog set.

## Current State Summary

MapSimulator is no longer "missing player simulation".
The project already has:

- A playable local character with foothold, ladder, rope, swim, fly, jump, prone, hit, death, and respawn states.
- Character WZ loading for body, head, face, hair, equipment, weapon overlays, z-map ordering, and slot-visibility conflicts.
- A working generic skill system with job skill loading, buffs, cooldowns, projectiles, hit effects, and quick-slot assignment.

The real parity gap is no longer "does the simulator have a feature at all?".
The gap is now "how closely does the current implementation match client behavior, data selection, timing, and fallback rules?".
